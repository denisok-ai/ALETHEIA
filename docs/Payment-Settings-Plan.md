# План: настройки оплаты (PayKeeper) в админке

**Цель:** вынести настройки подключения к платёжному сервису **PayKeeper** в отдельный блок раздела «Настройки» админки, вывести в интерфейс и в БД все поля, необходимые для подключения. Секретные значения хранить в БД в зашифрованном виде (как API-ключи в «Настройки AI») с fallback на переменные окружения.

**Сервис для оплаты:** PayKeeper. Документация: [help.paykeeper.ru](https://help.paykeeper.ru/), [JSON API — docs.paykeeper.ru](https://docs.paykeeper.ru/dokumentatsiya-json-api/).

---

## 1. Исследование подключения PayKeeper

### 1.1. Текущая интеграция в проекте

- **Файлы:** `lib/paykeeper.ts`, `app/api/payment/create/route.ts`, `app/api/webhook/paykeeper/route.ts`.
- **Источник настроек:** только переменные окружения (`.env`): `PAYKEEPER_SERVER`, `PAYKEEPER_LOGIN`, `PAYKEEPER_PASSWORD`, `PAYKEEPER_SECRET`.
- **Использование:** получение токена (Basic Auth: login + password), создание счёта (invoice), проверка подписи webhook по секретному слову (`secret` = informer_seed в терминологии PayKeeper).

### 1.2. Параметры подключения по документации PayKeeper

| Параметр | Назначение | Где задаётся в PayKeeper | Чувствительность |
|----------|------------|---------------------------|------------------|
| **Сервер (server)** | Хост личного кабинета PayKeeper, например `demo.server.paykeeper.ru` или `ваш-сервер.paykeeper.ru`. Без протокола. | Личный кабинет → адрес сервера | Нет (публичный) |
| **Логин (login)** | Идентификатор пользователя для Basic Auth при запросах к JSON API. | Личный кабинет → настройки доступа | Да |
| **Пароль (password)** | Пароль для Basic Auth (получение токена, создание счёта). | Там же | Да (секрет) |
| **Секретное слово (secret)** | Секрет для подписи уведомлений (informer_seed). Используется в MD5-подписи webhook: `id|sum|orderid|secret`. | Личный кабинет → настройки информера / приём POST-оповещений | Да (секрет) |

Дополнительно для продакшена:

| Параметр | Назначение |
|----------|------------|
| **URL уведомлений (webhook)** | URL нашего приложения, на который PayKeeper отправляет POST при смене статуса платежа. В нашем приложении: например `https://yourdomain.com/api/webhook/paykeeper`. Задаётся в личном кабинете PayKeeper. | В админке можно показывать «подставной» URL (текущий домен + путь) как подсказку для ввода в ЛК PayKeeper. |

Ссылки: [Приём POST оповещений](https://docs.paykeeper.ru/metody-integratsii/priyom-post-opoveshhenij/), [Модификация настроек](https://docs.paykeeper.ru/dokumentatsiya-json-api/modifikatsiya-nastroek/).

### 1.3. Сравнение с типовыми платёжными модулями (референс T-Bank / Tilda)

В интеграциях платёжных систем с CMS (например, [интеграция T-Bank с Tilda](https://developer.tbank.ru/eacq/modules/tilda)) обычно настраивают:

- **Идентификатор терминала (Terminal ID)** и **Пароль** — аналог нашей пары логин/пароль PayKeeper.
- **URL для уведомлений** — куда шлётся webhook (у нас — в ЛК PayKeeper).
- **Онлайн-касса, СНО, НДС** — для чеков; для PayKeeper при необходимости выносится отдельно.

Для **нашего сервиса PayKeeper** в админке достаточно полей: **сервер, логин, пароль, секрет**; URL уведомлений — справочно (подсказка для настройки в ЛК PayKeeper).

## 2. Поля для интерфейса и БД

Все перечисленные ниже ключи хранить в таблице **SystemSetting** (category: `payments`). Секретные значения (пароль, секрет) — в зашифрованном виде (как в разделе «Настройки AI» для API-ключей), ключ шифрования — `NEXTAUTH_SECRET`.

| Ключ в БД | Описание поля | Тип ввода в UI | Обязательность |
|-----------|----------------|----------------|----------------|
| `paykeeper_server` | Сервер PayKeeper (хост без https://), например `demo.server.paykeeper.ru` | Текстовое поле | Да |
| `paykeeper_login` | Логин (идентификатор пользователя API) | Текстовое поле | Да |
| `paykeeper_password` | Пароль для доступа к API | Пароль (маскирование), при сохранении — шифрование | Да |
| `paykeeper_secret` | Секретное слово для подписи webhook (informer_seed) | Пароль (маскирование), при сохранении — шифрование | Да |
| `paykeeper_webhook_url_hint` | Подсказка: URL для уведомлений в ЛК PayKeeper (только для отображения, например `{site_url}/api/webhook/paykeeper`) | Только чтение, вычисляется из site_url | Нет |

В форме админки:

- **Сервер** и **Логин** — обычные текстовые поля.
- **Пароль** и **Секрет** — поля типа password; при редактировании: пусто = «не менять текущее значение»; если введено новое — сохранять зашифрованно в БД. При чтении в API не отдавать значения, только флаг «задано» / «не задано».
- Блок **«URL для уведомлений»**: подпись вида «В личном кабинете PayKeeper укажите URL: https://ваш-домен/api/webhook/paykeeper» (подставлять site_url из настроек).

## 3. Логика чтения настроек (приложение)

- **Чтение:** при создании счёта и при приёме webhook брать конфиг PayKeeper из БД (расширить `lib/settings.ts` или добавить `getPayKeeperConfig()`). Если в БД значение пусто — fallback на `process.env.PAYKEEPER_*`.
- **Приоритет:** БД (расшифровка пароля/секрета при наличии) → env. Так администратор может задать всё в интерфейсе без правки .env.
- **Кеш:** кешировать конфиг на 1–2 минуты или сбрасывать кеш при сохранении настроек платежей в админке.

---

## 4. Интерфейс в админке

- **Раздел «Настройки»** → отдельная карточка **«Платежи (PayKeeper)»** (вынести из текущего текстового блока в полноценную форму).
- Заголовок: «Подключение PayKeeper».
- Подзаголовок: «Параметры для создания счетов и приёма уведомлений. Секретные поля хранятся в зашифрованном виде.»
- Поля формы:
  1. Сервер (обязательное)
  2. Логин (обязательное)
  3. Пароль (обязательное при первом сохранении; при редактировании — «оставить пустым, чтобы не менять»)
  4. Секретное слово для webhook (аналогично паролю)
  5. Блок только для чтения: «URL для уведомлений в ЛК PayKeeper» = `{site_url}/api/webhook/paykeeper`
- Кнопка «Сохранить»; при успехе — тост «Настройки PayKeeper сохранены».
- Ссылка на документацию: [help.paykeeper.ru](https://help.paykeeper.ru/), [Приём POST оповещений](https://docs.paykeeper.ru/metody-integratsii/priyom-post-opoveshhenij/).

## 5. API

- **GET /api/portal/admin/settings** — расширить ответ: для ключей `paykeeper_*` отдавать только наличие (например, `paykeeper_server`, `paykeeper_login` — значения; `paykeeper_password`, `paykeeper_secret` — флаги `true`/`false` без самих значений).
- **PATCH /api/portal/admin/settings** — принять ключи `paykeeper_server`, `paykeeper_login`, `paykeeper_password`, `paykeeper_secret`; пароль и секрет при непустом значении — шифровать перед записью в БД. Валидация: сервер и логин не пустые при сохранении.
- **Серверная логика** (`lib/paykeeper.ts`): новая функция `getPayKeeperConfigFromSettings()` (async), читает из БД/кеша, расшифровывает пароль и секрет; при отсутствии — использовать `getConfig()` из env. `createPayKeeperInvoice` и webhook вызывают единый источник конфигурации.

---

## 6. Безопасность

- Пароль и секрет в БД только в зашифрованном виде (AES-256-GCM или аналог, ключ из NEXTAUTH_SECRET).
- В API и в логах не выводить расшифрованные значения.
- Аудит: при изменении настроек PayKeeper записывать в AuditLog действие `settings.update`, entity `SystemSetting`, без сохранения самих секретов в diff.
