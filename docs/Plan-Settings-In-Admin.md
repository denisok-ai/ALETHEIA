# План: перенос настроек в меню админки и хранение в БД

**Цель:** вынести редактируемые параметры из конфигурационных файлов (.env, next.config и т.п.) в раздел «Настройки» админ-панели и хранить их в БД, сохраняя удобство и безопасность.

**Ограничение:** секреты (API-ключи, пароли, NEXTAUTH_SECRET) остаются только в переменных окружения; в UI отображаются лишь индикаторы «настроено / не настроено» без отображения значений.

---

## 1. Что переносим в БД, что остаётся в env

| Источник | Переносим в БД (редактируемо в UI) | Оставляем в env (только индикатор в UI) |
|----------|-------------------------------------|------------------------------------------|
| **.env** | `NEXT_PUBLIC_URL`, `RESEND_FROM`, `RESEND_NOTIFY_EMAIL`. **PayKeeper:** все поля подключения (сервер, логин, пароль, секрет) переносятся в БД с шифрованием секретов; env — fallback. См. **docs/Payment-Settings-Plan.md** | `NEXTAUTH_SECRET`, `RESEND_API_KEY`, `DEEPSEEK_API_KEY`, `TELEGRAM_BOT_TOKEN`, `DATABASE_URL` |
| **next.config** | При необходимости: `NEXT_PUBLIC_*`-переменные, которые решают «поведение» сайта (например, название портала) | Параметры сборки (images, experimental и т.д.) не трогаем |

**Идея:** в БД храним только то, что администратор может безопасно менять без доступа к серверу: URL сайта, email отправителя/получателя уведомлений, тексты, флаги функций. Секреты не показываем и не дублируем в БД.

---

## 2. Модель данных (БД)

- **Вариант A (рекомендуемый):** одна таблица `SystemSetting` с ключом и значением (как уже есть `LlmSetting` по сути). Расширяем использование или вводим отдельную сущность для «общих» настроек портала.

  Пример полей:
  - `key` (String, unique): например `site_url`, `resend_from`, `resend_notify_email`, `portal_title`, `contact_phone`.
  - `value` (String или Text): значение.
  - `category` (String, опционально): для группировки в UI (`general`, `email`, `payments`, `branding`).
  - `updatedAt` (DateTime).

- **Вариант B:** отдельная таблица `PortalConfig` с колонками под каждую настройку (менее гибко при добавлении новых полей).

Рекомендация: **Вариант A** (key-value с категорией), чтобы добавлять новые настройки без миграций схемы.

---

## 3. Структура меню «Настройки» с точки зрения дизайнера и UX/UI

### 3.1 Принципы

- **Группировка по смыслу:** не один длинный список, а блоки (карточки/секции) с заголовками и короткими подписями.
- **Прогрессивное раскрытие:** сначала видно только название группы и статус (например, «Почта: настроено»); по клику или «Редактировать» открывается форма полей этой группы.
- **Один экран — одна задача:** одна группа настроек = одна форма; сохранение с явной кнопкой «Сохранить» и фидбеком (успех/ошибка).
- **Безопасность и доверие:** секреты не показываем; для чувствительных полей (email) — валидация и подсказки; при смене критичных настроек — краткое предупреждение (например, «изменение URL повлияет на ссылки в письмах»).
- **Консистентность с порталом:** те же компоненты (Input, Label, Button, карточки), те же отступы и типографика, что в остальной админке.

### 3.2 Предлагаемая структура экрана «Настройки»

1. **Шапка**
   - Заголовок: «Настройки».
   - Краткий подзаголовок: «Параметры портала и интеграций. Секреты задаются в переменных окружения.»

2. **Блок «О системе»** (только чтение)
   - Версия приложения, окружение (development/production). Без редактирования.

3. **Блок «Состояние интеграций»** (только индикаторы)
   - Список интеграций (Почта, Telegram, AI, PayKeeper, NextAuth, БД) с индикатором «настроено» / «не задано» и подсказкой «значение в .env». Без отображения самих секретов.

4. **Редактируемые группы (из БД)**

   - **Общие**
     - URL сайта (NEXT_PUBLIC_URL → сохраняем в БД, приложение читает из БД с fallback на env).
     - Название портала (для шапки/подвала, если нужно).
     - Контактный телефон/email для отображения на сайте (если хранятся не только в контенте).
   - **Почта (отправка уведомлений)**
     - Email отправителя (RESEND_FROM).
     - Email получателя уведомлений (RESEND_NOTIFY_EMAIL).
     - Подпись: «API-ключ задаётся в .env (RESEND_API_KEY).»
   - **Платежи (PayKeeper)**
     - Отдельная форма: сервер, логин, пароль, секретное слово для webhook; подсказка URL уведомлений. Секреты хранятся в БД в зашифрованном виде, в UI — поля типа password (пусто = не менять). См. **docs/Payment-Settings-Plan.md**.
   - **Брендинг и тексты** (опционально)
     - Тексты для писем, подписи, название школы в рассылках — если решим вынести в настройки.

Каждая группа:
- Карточка с заголовком и коротким описанием.
- Кнопка «Изменить» или сразу форма (в зависимости от количества полей).
- Поля с подписями и placeholder’ами.
- Кнопка «Сохранить»; после успеха — тост «Настройки сохранены» и при необходимости обновление индикаторов.

5. **Блок «Переменные окружения»**
   - Справочный список переменных (как сейчас): название, описание, без значений. Ссылка на документацию (Deploy.md, .env.example).

6. **Документация**
   - Ссылки на docs (Deploy, Local-Prisma и т.д.).

### 3.3 Визуальная иерархия

- Заголовки секций: один уровень (например, `text-lg font-semibold`).
- Подзаголовки/описания: приглушённый цвет, меньший кегль.
- Поля формы: label сверху, input под ним; между группами полей — отступ.
- Кнопки: основной акцент у «Сохранить», вторичный стиль у «Отмена» / «Изменить».
- Индикаторы интеграций: иконка или цветовая точка (зелёный/красный) + короткий текст.

### 3.4 Адаптивность и доступность

- На планшетах и мобильных блоки идут колонкой; формы не перегружены.
- Кнопки и поля достаточного размера для тапов.
- Сообщения об ошибках валидации — рядом с полем и в общем блоке при ошибке API.
- Для критичных действий (например, смена URL) — дополнительное подтверждение (модалка или второй шаг).

## 4. Технические шаги реализации

1. **Схема БД:** таблица `SystemSetting` (key, value, category?, updatedAt) или расширение существующего хранилища ключ–значение.
2. **API:** `GET /api/portal/admin/settings` — отдать все настройки по категориям; `PATCH /api/portal/admin/settings` — обновить переданные ключи (только разрешённый список ключей).
3. **Чтение в приложении:** хелпер или сервис, который при старте/по запросу читает настройки из БД и отдаёт их (с кешем на несколько минут или при первом запросе), с fallback на process.env для обратной совместимости.
4. **Страница настроек:** разбить на компоненты по блокам (Общие, Почта, Платежи, Справочник env, Документация); формы с сохранением через PATCH; индикаторы интеграций через существующий env-check.
5. **Валидация:** URL — формат; email — формат; обязательность полей в зависимости от бизнес-правил.
6. **Аудит:** запись в AuditLog при изменении настроек (action типа `settings.update`, entity `SystemSetting`).

---

## 5. Порядок внедрения

| Шаг | Действие |
|-----|----------|
| 1 | Ввести модель SystemSetting и миграцию; API GET/PATCH с белым списком ключей. |
| 2 | Реализовать чтение настроек в коде (хелпер + fallback на env). |
| 3 | Перевести текущие «только справочные» блоки на странице в описанную структуру; добавить блок «Общие» и «Почта» с формами. |
| 4 | Подключить использование настроек из БД для RESEND_FROM, RESEND_NOTIFY_EMAIL, NEXT_PUBLIC_URL (или аналога) в коде отправки писем и генерации ссылок. |
| 5 | Добавить аудит изменений и при необходимости подтверждение для критичных полей. |

---

## 6. Риски и упрощения

- **Кеш:** чтобы не дергать БД при каждом запросе, настройки можно кешировать в памяти (TTL 1–5 мин) или при старте приложения.
- **Секреты:** ни при каких условиях не выводить и не сохранять в БД; проверки в API (белый список ключей) не допускают записи полей типа `RESEND_API_KEY`.
- **Миграция с env:** при первом запуске можно заполнить SystemSetting значениями из env для ключей site_url, resend_from, resend_notify_email, чтобы не ломать текущее поведение.

После реализации этого плана меню «Настройки» станет единой точкой управления редактируемыми параметрами портала с понятной структурой и без риска случайно показать секреты в UI.
