# План разработки модуля «Мониторинг»

Документ составлен на основе мини-ТЗ и инструкции профессиональной системы Mirapolis [1–31]. Модуль предназначен для оперативного контроля состояния системы, анализа нагрузки и истории посещений.

---

## 1. Назначение модуля

**Цель:** оперативный контроль состояния системы, анализ текущей нагрузки и просмотр истории посещений пользователей.

**Аудитория:** администратор портала (при необходимости — менеджер с доступом к мониторингу).

**Референс:** экраны Mirapolis — «Пользователи Online», «Посещения» (статистика + график), «Выполняемые задачи», «Системное окружение пользователя».

---

## 2. Функциональные блоки (MVP)

### 2.1. Пользователи Online

| Требование | Описание |
|------------|----------|
| Сводка по ролям | Таблица: **Профиль (роль)** \| **В системе** (количество активных сессий). Роли: Администратор, Слушатель, Менеджер и т.д. Действие «Подробнее» — переход к списку пользователей данной роли. |
| Детальный список | Таблица: №, **Пользователь** (ФИО — ссылка на карточку пользователя), **Профиль**, **Время входа**, **Последний запрос** (время последнего запроса к серверу), **IP-адрес**. Поиск «Найти в списке». Пагинация. |
| Определение Online | Пользователь считается в системе, если сессия активна и время последнего запроса не превышает заданный таймаут (например, 15–30 мин). |

**Защита данных (из источников):** ссылка на ФИО в списках мониторинга ведёт на карточку пользователя («Физическое лицо») — в нашей системе: `/portal/admin/users/[id]`.

---

### 2.2. Статистика посещений (Посещения)

| Требование | Описание |
|------------|----------|
| Период | Выбор периода «с … по …» (дата входа/выхода). Кнопки «Обновить», «Очистить». |
| Вкладка «Статистика» | Таблица: №, **Пользователь** (ссылка на карточку), **Количество посещений** (сессий за период). Поиск по имени. Пагинация. Действие «Подробнее» / «Время посещения» — детализация по пользователю. |
| Детализация по пользователю | Экран «Время посещения»: список сессий выбранного пользователя за период. Колонки: IP-адрес, Время входа, Время выхода. Опция «Окружение» — системное окружение сессии (UserAgent, ОС, браузер и т.п.). |
| Вкладка «График» | **Гистограмма:** по горизонтали — дни выбранного месяца (1–28/29/30/31), по вертикали — количество **уникальных посетителей** в каждый день. Выбор года и месяца, кнопка «Построить». |
| Очистка логов | Возможность очистки: «Очистить всё» или «Очистить старше 1 месяца» (или настраиваемый порог). |

**Как построить график (по инструкции):** Перейти в Мониторинг → Посещения → вкладка «График» → выбрать год и месяц → выполнить действие «Построить». На графике отображается количество уникальных посетителей по дням месяца.

---

### 2.3. Выполняемые задачи (Active Tasks)

| Требование | Описание |
|------------|----------|
| Список процессов | Таблица фоновых задач, запущенных в данный момент (рассылки, генерация отчётов, массовая выдача сертификатов и т.п.). |
| Атрибуты задачи | Название, инициатор (пользователь), **прогресс (%)**, время старта. Для мини-системы критично отображать прогресс в процентах, чтобы видеть, не «зависла» ли задача. |
| Действие | Кнопка «Прервать» с предподтверждением — принудительное прерывание задачи администратором. |

---

### 2.4. Системное окружение пользователя (опционально для MVP)

При детализации сессии — экран «Системное окружение пользователя»: пользователь, время входа, время выхода; таблица параметров (ОС, браузер, разрешение экрана, JavaScript, Cookies и т.д.). Данные сохраняются при входе/запросах (UserAgent, опционально — разбор в структурированные поля).

## 3. Логика учёта данных (Data Tracking)

| Данные | Когда фиксировать | Где хранить |
|--------|-------------------|-------------|
| **lastActivityAt** | На каждом запросе аутентифицированного пользователя (middleware или хук в сессии). | Сессия (NextAuth/Session) или отдельная таблица `UserSession` / кэш. |
| **loginTime** | В момент входа (signIn callback, после проверки пароля). | Таблица сессий или `VisitLog`. |
| **logoutTime** | В момент выхода (signOut) или по таймауту неактивности. | Та же таблица. |
| **IPAddress** | Из заголовка запроса (X-Forwarded-For, X-Real-IP или request.socket). | В записи сессии/посещения. |
| **UserAgent** | Из заголовка User-Agent. | В записи сессии; опционально — разбор (браузер, ОС) для экрана «Системное окружение». |

**Модель данных (предложение):**

- **UserSession** (или расширение сессии): `id`, `userId`, `sessionToken` (если NextAuth), `loginAt`, `lastActivityAt`, `logoutAt` (nullable), `ipAddress`, `userAgent` (текст или JSON). Индекс по `userId`, `lastActivityAt`.
- **VisitLog** (альтернатива или дополнение): то же для истории посещений с возможностью очистки по дате.

Определение **Online:** сессия считается активной, если `logoutAt` пусто и `lastActivityAt` в пределах таймаута (например, последние 15 минут).

## 4. API (эскиз)

| Метод | Маршрут | Назначение |
|-------|---------|------------|
| GET | `/api/portal/admin/monitoring/online` | Сводка по ролям + список активных сессий (с пагинацией, поиском). |
| GET | `/api/portal/admin/monitoring/visits` | История посещений: список пользователей и количество сессий за период; фильтр по имени. |
| GET | `/api/portal/admin/monitoring/visits/chart` | Данные для графика: год, месяц → массив { day, uniqueVisitors }. |
| GET | `/api/portal/admin/monitoring/visits/user/[userId]` | Детализация по пользователю: список сессий (IP, время входа, время выхода) за период. |
| POST | `/api/portal/admin/monitoring/visits/clear` | Очистка логов: body `{ olderThanDays?: number }` или «всё». |
| GET | `/api/portal/admin/monitoring/tasks` | Список выполняемых фоновых задач (название, инициатор, прогресс %, время старта). |
| POST | `/api/portal/admin/monitoring/tasks/[taskId]/interrupt` | Прервать задачу (с проверкой прав). |

Обновление **lastActivityAt:** в middleware для защищённых маршрутов (или в обёртке getServerSession) — раз в N минут обновлять запись в БД/кэше по текущей сессии.

## 5. Интерфейс (UI)

- **Раздел «Мониторинг»** в меню админки (иконка Activity или BarChart2).
- **Три вкладки:** «Пользователи Online» | «Посещения» | «Выполняемые задачи».
- **Пользователи Online:** карточка со сводной таблицей по ролям; ниже — таблица с детальным списком (ФИО — ссылка на `/portal/admin/users/[id]`), время входа, последний запрос, IP. Поиск, пагинация. Опционально: автообновление списка (polling раз в 30–60 сек).
- **Посещения:** период «с … по …», кнопки Обновить/Очистить. Подвкладки: **Статистика** (таблица пользователь → кол-во посещений; переход к детализации по пользователю) и **График** (выбор год/месяц, кнопка «Построить», гистограмма: дни месяца → уникальные посетители). Поиск по имени в таблице статистики.
- **Выполняемые задачи:** таблица (название, инициатор, прогресс %, время старта), кнопка «Прервать» с диалогом подтверждения.
- Стиль: светлые карточки, иконки Lucide (Activity, Users, Clock), кнопки в стиле золотой/primary темы LMS.

## 6. Этапы внедрения

| Этап | Задача | Приоритет |
|------|--------|-----------|
| 1 | Модель сессий/посещений (UserSession или VisitLog), миграция; middleware обновления lastActivityAt; фиксация login/logout, IP, UserAgent | Высокий |
| 2 | API: online (сводка + список), visits (таблица + chart), clear | Высокий |
| 3 | Страница «Мониторинг»: вкладки «Пользователи Online» и «Посещения» (статистика + график), ссылки на карточки пользователей | Высокий |
| 4 | Детализация «Время посещения» по пользователю; опционально — «Системное окружение» | Средний |
| 5 | Модель/реестр фоновых задач, API tasks + interrupt; вкладка «Выполняемые задачи» и кнопка «Прервать» | Средний |
| 6 | Очистка логов (кнопка «Очистить» с выбором «всё» / «старше N дней») | Средний |
| 7 | Автообновление списка Online (polling) | Низкий |
