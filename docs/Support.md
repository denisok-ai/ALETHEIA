## Частые задачи

### Оптимизация изображений

Логотип и превью курсов используют `next/image` (автоматические webp/avif, lazy load).

### Изменить тексты на главной

- Секции: `components/sections/` (Hero, About, Pricing, Contact и т.д.)
- Контент: `docs/Content.md` (референс)

### Новости и публикации

- **ID публикаций** — строковые (nanoid), не числа. Ссылки формируются из API `GET /api/publications`; пример: `/news/cmmroxynh00fk9t6ozd3patpt`. `/news/1` даёт 404.
- **Карточка:** `app/news/[id]/page.tsx`.

### Оферта и политика конфиденциальности

- Тексты: `app/oferta/page.tsx`, `app/privacy/page.tsx`. Метаданные (title, description) заданы в тех же файлах. Sitemap и robots: `app/sitemap.ts`, `app/robots.ts` (базовый URL из NEXT_PUBLIC_URL).

### Добавить/изменить тариф (товар для продажи)

- **Магазин на главной:** товары подгружаются из GET `/api/shop/products` (активные Service с привязанным опубликованным курсом). При отсутствии товаров показывается резервный список из `components/sections/Pricing.tsx` (FALLBACK_TARIFFS).
- **Админка:** раздел «Товары для продажи на главной» на странице `/portal/admin/payments` — добавление/редактирование услуг (slug, название, цена, ID тарифа PayKeeper, привязка к курсу). Создание заказа: POST `/api/payment/create` с `serviceSlug` или `tariffId`; таблица `Service` в Prisma (paykeeperTariffId → courseId), webhook создаёт запись на курс.
- **Тестовое подключение PayKeeper:** в Настройки → Платежи — чекбокс «Использовать тестовое подключение» и отдельные поля (сервер, логин, пароль, секрет). Env: `PAYKEEPER_USE_TEST=1`, `PAYKEEPER_TEST_*`.
- **Страница успешной оплаты:** `/success` — после оплаты покупатель попадает сюда автоматически (при создании счёта передаётся `user_result_callback` = site_url + `/success?order=НОМЕР_ЗАКАЗА`). Для гостя при наличии `?order=…` показывается «Заказ № … оплачен» и маскированный email; для авторизованных — «Ваш курс уже в Мои курсы». В Настройки → Платежи выведен URL успешной оплаты.
- **Письма после оплаты:** темы и тела писем (с курсом и без курса) настраиваются в **Настройки → Шаблоны писем об оплате**. Плейсхолдеры: `{{orderid}}`, `{{courseTitle}}`, `{{loginUrl}}`, `{{successUrl}}`, `{{portal_title}}`. При пустых полях используются тексты по умолчанию. Отправка через Resend (RESEND_API_KEY).
- **Связь лид ↔ заказ:** при оплате (webhook) у лидов с тем же email проставляется поле «Оплаченный заказ» (Lead.lastOrderNumber). В CRM в карточке лида и в экспорте CSV отображается номер заказа.
- **Тикет «Нет доступа»:** при создании обращения в поддержку система пытается привязать оплаченные заказы (claim). Если найден оплаченный заказ с курсом без доступа у пользователя, тикету задаётся тема «Не приходит доступ — …» и привязывается номер заказа (Ticket.orderNumber). В письме менеджеру и в интерфейсе тикета отображается «Заказ: …» со ссылкой на раздел Оплаты (поиск по номеру).

### Настройки PayKeeper: где хранятся

Параметры PayKeeper (сервер, логин, пароль, секрет для webhook) задаются в **Настройки → Платежи** и сохраняются в БД (секреты — в зашифрованном виде). Приложение при создании счёта и при приёме webhook сначала читает конфиг из БД (`getPayKeeperConfigFromSettings`); если в БД ничего нет или не хватает полей — используется fallback на переменные окружения (.env). Для симуляции оплаты **без реального PayKeeper** настройки в админке не обязательны: блок «Симуляция оплаты» создаёт заказ и обрабатывает его на стороне сервера без подписи webhook.

### Проверка сценария «оплата → доступ к курсу»

**Рекомендуемый способ — симуляция из админки (без PayKeeper и без .env)**

Все шаги выполняются в браузере, секрет из настроек не нужен.

1. **Запустите приложение** (например `npm run dev` или `npm run dev:debug:seed`). Убедитесь, что в БД есть тестовые пользователи и курсы (после `npx prisma db seed` есть студент `student1@test.local` с паролем `Test123!` и товары с slug `course-1`, `course-2` и т.д.).

2. **Войдите в админку** под учётной записью администратора (например `admin@test.local` / `Test123!`).

3. **Откройте раздел «Оплаты»:** Портал → Оплаты (`/portal/admin/payments`).

4. **Проверьте блок «Товары для продажи на главной».** Должен быть хотя бы один активный товар с привязанным курсом (slug, название, цена, ID тарифа PayKeeper, курс). Если товаров нет — нажмите «Добавить товар», укажите slug (например `course-1`), название, цену, привяжите курс и сохраните.

5. **В блоке «Симуляция оплаты (без PayKeeper)»** укажите:
   - **Товар (услуга)** — выберите из списка товар с привязанным курсом (например «Тариф «Курс «Тело не врет»…» / `course-1`).
   - **Email пользователя** — email существующего пользователя, которому нужно «выдать» доступ (например `student1@test.local`).
   - **Имя** — по желанию (по умолчанию «Тест»).

6. **Нажмите «Симулировать оплату».** Должно появиться уведомление об успехе и номер заказа. В таблице заказов ниже появится новый заказ со статусом «Оплачен».

7. **Проверьте доступ у студента:** выйдите из админки (или откройте портал в режиме инкогнито/другом браузере), войдите под указанным email (`student1@test.local` / `Test123!`), откройте **Мои курсы**. Курс, привязанный к выбранному товару, должен отображаться в списке.

Итог: симуляция использует только данные из БД (товары, курсы, пользователи, шаблоны писем из Настроек). Настройки PayKeeper в панели для этого сценария не требуются.

**Вариант с реальным тестовым PayKeeper**

1. В **Настройки → Платежи** включите «Использовать тестовое подключение» и укажите тестовые сервер, логин, пароль и секрет (или задайте в .env `PAYKEEPER_USE_TEST=1` и `PAYKEEPER_TEST_*`).
2. Убедитесь, что в разделе **Оплаты** есть активный товар с привязкой к курсу и ID тарифа, совпадающий с тарифом в ЛК PayKeeper.
3. На главной сайта в блоке «Купить курс» выберите товар, в модалке укажите email существующего студента, имя → «Перейти к оплате».
4. В тестовом PayKeeper завершите оплату. После вызова webhook PayKeeper перенаправит на `/success?order=…`.
5. Войдите под этим студентом → **Мои курсы** — курс должен появиться.

**Симуляция через скрипт (опционально)**

Если нужна проверка именно вызова webhook с подписью (например, для отладки), можно использовать скрипт `scripts/simulate-paykeeper-webhook.sh`. Для подписи нужен секрет: он берётся из **переменной окружения** `PAYKEEPER_SECRET` (или `PAYKEEPER_TEST_SECRET` при тестовом режиме). Если вы храните секрет только в админке (БД), скрипт не подойдёт — используйте блок «Симуляция оплаты» в разделе Оплаты.

**Важно:** Webhook ищет услугу по полю `paykeeperTariffId` заказа: при создании заказа по товару с slug `course-1` в Order сохраняется `tariffId`, равный `paykeeperTariffId` этой услуги (в seed это `tariff-1`). По нему находится курс и создаётся запись на курс для пользователя с email из заказа.

### Настроить чат-бота

- Всё в разделе **Настройки AI** (`/portal/admin/ai-settings`): ключ API моделей LLM (хранится в БД), шаблоны промптов (один активный, статистика, генерация с AI), база знаний по продукту (редактируется в интерфейсе, хранится в БД), параметры чат-бота и тьютора, тестовый чат.
- Резерв: база знаний при первом открытии подтягивается из `content/knowledge-base-body-never-lies.md`, если в БД пусто.
- API чата: `app/api/chat/route.ts` (ключ и база знаний из БД).

### Установка пароля после конвертации лида

После конвертации лида в пользователя (CRM → «Конвертировать в пользователя») клиенту на email отправляется письмо со ссылкой **Установить пароль**. Ссылка ведёт на `/set-password?token=…`, действует 48 часов. После установки пароля пользователь входит через обычную страницу входа. Токены хранятся в таблице `PasswordToken`.

### Сброс пароля

Пользователь на странице входа нажимает «Забыли пароль?» → переходит на `/reset-password`, вводит email → POST `/api/auth/forgot-password` создаёт одноразовый токен (таблица `PasswordToken`, 48 ч) и отправляет письмо со ссылкой на `/set-password?token=…`. Установка нового пароля — та же страница и API, что и при конвертации лида. Для защиты от перебора ответ API всегда «успех» (не раскрывает наличие email). Админ может сбросить пароль вручную: карточка пользователя → вкладка «Профиль» → «Сбросить пароль» (отправка ссылки на set-password пользователю, если реализовано в админке).

### Несохранённые изменения

В формах редактирования **шаблона уведомления** и **шаблона сертификата** при изменении полей и попытке уйти со страницы (кнопка «К списку»/«Отмена») или закрыть вкладку/обновить страницу показывается предупреждение о несохранённых изменениях. Используется хук `lib/useUnsavedChanges(isDirty)`.

### Прогресс SCORM

Прогресс прохождения курсов сохраняется в таблице `ScormProgress` (поля: lessonId, completionStatus, score, timeSpent, cmiData). API: GET/POST `/api/portal/scorm/progress` (один урок), GET `/api/portal/scorm/progress/all?courseId=` (все уроки курса). Статусы «completed» и «passed» считаются завершением урока; при 100% завершения по курсу автоматически выставляется сертификат. На карточке курса и на странице курса студента отображаются: процент, «X из Y уроков», время, средний балл (если пакет передаёт score). В плеере прогресс обновляется после каждого успешного коммита и раз в 15 сек.

### Проверка SCORM-плеера

Курсы без загруженного SCORM показывают «У этого курса пока нет загруженного SCORM-контента». Для проверки плеера:

1. **После seed:** `npx prisma db seed` создаёт минимальный SCORM-контент (index.html, lesson1–3.html) для `course-seed-1` и `course-seed-2` в `public/uploads/scorm/courses-{id}/`. Студент с доступом к курсу увидит 3 урока.
2. **Загрузить SCORM через админку:** Курсы → выбрать курс → «Загрузить SCORM». Загрузить ZIP (например, `docs/Agile (7).zip` для E2E).
3. **E2E-тесты:** используют `course-seed-2` и `docs/Agile (7).zip` — загрузка через админку перезаписывает минимальный контент из seed.
