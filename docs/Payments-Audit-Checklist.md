# Чек-лист: оплаты и письма (кросс-ролевой аудит)

Использовать перед продакшеном и после смены цен/тарифов. Цель: согласованность статусов, текстов и ссылок между админкой, ЛК студента, почтой и публичным сайтом.

## Администратор

- [ ] **Портал → Настройки:** PayKeeper (боевой или тестовый контур), URL сайта, почта Resend, шаблоны писем об оплате с превью и тестовой отправкой.
- [ ] **Товары (Service):** slug, цена, `paykeeperTariffId` совпадает с ЛК PayKeeper.
- [ ] **Webhook:** URL `POST /api/webhook/paykeeper` доступен по HTTPS; в логах сервера видны строки `[PayKeeper webhook] ok` при тесте (без утечки PII).
- [ ] Повторный callback по уже оплаченному заказу не дублирует письма (идемпотентность в `processPaidOrder`).

## Студент

- [ ] Оплата с публичной страницы → редирект success/fail понятен человеку, не только код ошибки.
- [ ] После успешной оплаты курс появляется в **Мои курсы** (при совпадении email).
- [ ] Текст на экране успеха не противоречит письму на почту.

## Менеджер

- [ ] По тикету клиента можно сопоставить email с заказом (админка пользователей / заказы) или дать корректную инструкцию без обещаний вне оферты.

## Гость / маркетинг

- [ ] Цена на лендинге курса и в модалке оплаты совпадает с записью `Service`.
- [ ] Оферта `/oferta` открывается; формулировки писем ссылаются на оферту.

## Инварианты

- [ ] Статусы заказа в админке и у студента описаны одинаковыми терминами (pending / paid / failed и т.д.).
- [ ] В URL и клиентских логах нет утечки полных персональных данных при ошибке оплаты.

## Staging

- [ ] Тестовый платёж PayKeeper → callback → статус `paid` в БД → одно письмо клиенту.
