# План доработок админ-панели AVATERRA

**Дата:** 2025-03-10  
**Цель:** Полный аудит функционала и связей, унификация UI/UX по образцу карточки курса, редизайн карточек и пользовательского пути, добавление недостающего функционала.

---

## 1. Аудит функционала и связей

### 1.1 Маршруты админки

| Маршрут | Назначение | Ключевые сущности |
|---------|------------|-------------------|
| `/portal/admin/dashboard` | Дашборд: метрики, графики, последние события | Order, Enrollment, Lead, Ticket, User, Course, Certificate |
| `/portal/admin/users` | Каталог пользователей | User, Profile |
| `/portal/admin/users/[id]` | Карточка пользователя | User, Profile, Enrollment, Certificate, Order, Ticket |
| `/portal/admin/courses` | Список курсов | Course |
| `/portal/admin/courses/[courseId]` | Карточка курса (табы) | Course, Enrollment, ScormProgress, Certificate, CourseNotificationSet, Media |
| `/portal/admin/courses/[courseId]/enrollments/[userId]` | Результаты одного участника | Enrollment, ScormProgress |
| `/portal/admin/payments` | Оплаты и заказы | Order |
| `/portal/admin/certificates` | Каталог сертификатов | Certificate, Course, User |
| `/portal/admin/media` | Медиатека | Media, Course |
| `/portal/admin/crm` | Лиды и воронка | Lead |
| `/portal/admin/communications` | Шаблоны и рассылки | CommsTemplate, CommsSend |
| `/portal/admin/ai-settings` | Настройки AI (чат-бот, тьютор) | LlmSetting |
| `/portal/admin/audit` | Журнал аудита | AuditLog |
| `/portal/admin/settings` | Системные настройки | SystemSetting |
| `/portal/admin/notification-sets/[id]` | Карточка набора уведомлений | NotificationSet |

### 1.2 Связи сущностей (кратко)

- **Course** ↔ Enrollment, Certificate, ScormProgress, Media, CourseNotificationSet, Service  
- **User** ↔ Profile, Enrollment, Certificate, Order, Ticket, Notification  
- **Order** → User (опционально), оплата → зачисление на курс  
- **Lead** → User (при конвертации)  
- **Enrollment** → Course, User; доступ (accessClosed), завершённость (completedAt)  
- **Certificate** → User, Course  
- **NotificationSet** ↔ Course (через CourseNotificationSet)  
- **Media** ↔ Course (многие-ко-многим через поле courseId в Media)

### 1.3 Эталон: паттерны карточки курса

Использовать как образец для остальных разделов:

| Элемент | Реализация в карточке курса |

| **Фильтры/поиск** | «Найти в списке» (поиск), фильтр по источнику/статусу где нужно |

## 3. Редизайн админки: UI/UX и пользовательский путь

### 3.1 Общие принципы

1. **Единая шапка страницы**  
   На каждой странице: Breadcrumbs → одна зона с заголовком (H1), подзаголовком (описание) и основными действиями (кнопки «Добавить», «Экспорт», «К списку» и т.д.) в одной строке/блоке.

2. **Карточки контента**  
   Все блоки контента (таблицы, формы, метрики) — в карточках:  
   `rounded-xl border border-border bg-white p-4` (или p-6 для крупных блоков).  
   Внутри: заголовок H2 (`text-lg font-semibold text-dark`), при необходимости краткое описание (`text-sm text-text-muted`).

3. **Таблицы**  
   Единый стиль:  
   - колонка № (где уместно);  
   - заголовки TableHead;  
   - чередование или hover для строк;  
   - действия в строке: ссылки (Подробнее, Название) + кнопки (Изменить, Удалить, Исключить) с одинаковыми отступами и размером.

4. **Панели действий над таблицей**  
   Слева: dropdown «Добавить» (если есть), фильтры (select), поле «Найти в списке».  
   Справа при необходимости: экспорт, «Для выбранных» (dropdown).  
   Всё в одной строке с flex-wrap.

5. **Пагинация**  
   Единый формат: кнопки размера страницы (+5, +10, +50), текст «Записи X–Y из Z», переключатель «Страница N из M» с стрелками.

6. **Пустые состояния**  
   Вместо «Нет данных» — короткий текст с контекстом: «Нет прикреплённых наборов уведомлений», «Нет записей на курсы» и т.д. При необходимости — иллюстрация или иконка (компонент EmptyState).

7. **Подтверждение опасных действий**  
   Удаление, отзыв, исключение, сброс пароля — всегда через ConfirmDialog с явным текстом и variant="danger".

8. **Пользовательский путь**  
   - Список → карточка объекта → вложенные сущности (табы или секции).  
   - Из карточки — всегда явная кнопка/ссылка «К списку» (или «К курсам», «К пользователям»).  
   - Breadcrumbs отражают путь: Дашборд → Раздел → Объект (при карточке).

### 3.2 Структура страницы (шаблон)

```
[Breadcrumbs]
[Блок заголовка: H1, описание, действия (кнопки)]
[Карточка 1: метрики или основная информация]
[Карточка 2: табы или одна таблица/форма]
  [Таб 1] [Таб 2] ...
  [Контент активного таба]
[При необходимости: ещё карточки]
[Пагинация под таблицей]
```

### 3.3 Сайдбар и навигация

- Сохранить текущий PortalSidebar (иконки, сворачивание).  
- Подсветка активного раздела: уже есть (pathname).  
- При редизайне проверить контраст и размер кликабельной области.

### 3.4 Цвета и типографика

- Следовать .cursorrules: primary #2D1B4E, secondary #D4AF37, dark #0A0E27.  
- Заголовки: font-heading (Literata), остальной текст — font-body (Outfit).  
- Карточки: белый фон, border-border.  
- Кнопки опасных действий: variant="danger" или явный красный акцент.

## 5. Детальный план доработок (очередность)

### Фаза 1: Унификация навигации и шапок (1–2 дня)

1. Ввести переиспользуемый компонент **PageHeader**: Breadcrumbs + H1 + описание + слот для действий (кнопки справа).  
2. Заменить текущие разрозненные Breadcrumbs + заголовки на PageHeader на: dashboard, users, users/[id], courses, courses/[courseId], payments, certificates, media, crm, communications, ai-settings, audit, settings.  
3. На всех карточках объекта (users/[id], courses/[courseId], notification-sets/[id]) добавить явную кнопку/ссылку «К списку» в блок действий PageHeader.

### Фаза 2: Карточки и секции (1–2 дня)

4. Задать единый класс карточки (например, утилита или компонент **Card**): `rounded-xl border border-border bg-white p-4`.  
5. На страницах dashboard, payments, crm, ai-settings, settings, audit — обернуть секции в карточки с единым стилем.  
6. Карточка пользователя [id]: разбить на табы (Профиль, Записи на курсы, Сертификаты, Заказы, Тикеты) по образцу курса.

### Фаза 3: Таблицы и действия (2–3 дня)

7. Во всех таблицах админки: добавить колонку № где уместно; унифицировать стиль кнопок в строках (размер, отступы).  
8. Пагинация: где её нет (media, audit, certificates при большом объёме) — добавить в формате +5/+10/+50 и «Страница N из M».  
9. Поиск «Найти в списке»: добавить на страницы media, communications (шаблоны), при необходимости — audit.  
10. ConfirmDialog: проверить все опасные действия (удаление, отзыв, исключение, конвертация лида) и единообразие текстов.

### Фаза 4: Пустые состояния и загрузка (0.5–1 день)

11. Ввести компонент **EmptyState** (иконка, заголовок, описание, опционально CTA).  
12. Заменить текстовые «Нет записей» на EmptyState в таблицах: курсы (пустой список), пользователи, оплаты, сертификаты, медиа, CRM, коммуникации, аудит, табы в карточке курса и пользователя.  
13. Убедиться, что при загрузке данных таблицы показывают TableSkeleton.

### Фаза 5: Недостающий функционал (по приоритету)

14. Карточка заказа (payments): страница /portal/admin/payments/orders/[id] или модальное окно с деталями заказа, ссылкой на пользователя, кнопками Подтвердить/Отменить.  
15. Опционально: форма «Добавить пользователя» в разделе Пользователи.  
16. Опционально: глобальный каталог наборов уведомлений /portal/admin/notification-sets (список всех наборов с переходом в карточку).

### Фаза 6: Полировка и доступность (0.5–1 день)

17. Проверить aria-label у кнопок и ссылок действий.  
18. Проверить контраст и фокус (клавиатурная навигация).  
19. Обновить docs/Support.md или внутреннюю документацию админки перечнем разделов и типовых действий.
