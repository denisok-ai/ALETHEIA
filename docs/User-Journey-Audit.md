# Аудит пользовательского пути: от консультации и покупки до авторегистрации и автоподдержки

**Дата:** 2026-03-10  
**Цель:** полный аудит сценариев от первого касания (заявка/покупка) до входа в ЛК и поддержки; выявление пробелов и план доработок.

---

## 1. Текущие потоки (как есть)

### 1.1 Заявка на консультацию / обратная связь

| Шаг | Реализация | Где |
|-----|------------|-----|
| Пользователь заполняет форму (имя, телефон, email, сообщение) | ✅ | Секция «Контакты», `Contact.tsx` → POST `/api/contact` |
| Валидация (honeypot, телефон ≥10 цифр) | ✅ | `api/contact/route.ts` |
| Запись в БД | ✅ | `Lead` (name, phone, email, message) |
| Уведомление администратору | ✅ | Resend → письмо на `resend_notify_email` |
| Связь заявки с последующей оплатой | ❌ | Нет (лид и заказ не связаны) |
| Ответ клиенту «Заявка принята» | ❌ | Клиенту письмо не отправляется |
| CTA «Оплатить консультацию» в письме/на сайте после заявки | ❌ | Нет |

**Итог:** заявка сохраняется и уходит админу; клиент не получает подтверждения и не получает явного перехода к оплате.

### 1.2 Покупка (оформление платежа)

| Шаг | Реализация | Где |
|-----|------------|-----|
| Выбор тарифа на главной (консультация, тренинг, курс, онлайн) | ✅ | `Pricing.tsx` + `/api/shop/products` или fallback `FALLBACK_TARIFFS` |
| Модалка «Оформление» (email, имя, телефон) | ✅ | `PaymentModal.tsx` |
| Создание заказа | ✅ | POST `/api/payment/create` → `Order` (pending), `orderNumber`, `clientEmail`, `clientPhone` |
| Редирект на PayKeeper | ✅ | `paymentUrl` из `createPayKeeperInvoice` |
| Success URL после оплаты | ✅ | `successRedirectUrl` = `{site}/success` (в инвойсе) |

**Итог:** поток покупки до редиректа на платёжную систему реализован.

### 1.3 Webhook PayKeeper (после оплаты)

| Условие | Действие | Проблемы |
|---------|----------|----------|
| Заказ найден | `Order.status = 'paid'`, `paidAt` | — |
| Есть `Service` с `courseId` по `tariffId` | Ищем `User` по `order.clientEmail` | — |
| **User найден** | `Enrollment` upsert, `triggerNotification('enrollment')`, `Order.userId` = user.id, письмо «Оплата получена — доступ к курсу открыт» | ✅ |
| **User не найден** | Письмо то же: «зарегистрируйтесь с этим email, и курс будет доступен» | ❌ Регистрация потом не привязывает заказ к пользователю (см. ниже) |
| Нет курса у тарифа (консультация, тренинг, онлайн) | Только обновление `Order.status` | ❌ Клиенту **вообще не отправляется письмо** об успешной оплате |

**Итог:** при оплате курса и существующем пользователе всё ок; при первом покупке курса и при оплате консультации/тренинга без курса — пробелы.

## 2. Сценарии (полный перечень)

### Сценарий A: Заявка на консультацию

1. Пользователь оставляет заявку (имя, телефон, email, сообщение).
2. **Ожидание:** клиент получает письмо «Заявка принята, мы свяжемся с вами»; админ видит лид в CRM и письмо.
3. **Сейчас:** клиент не получает подтверждения; админ получает письмо и видит лид.
4. **Пробел:** нет подтверждения клиенту; нет явного CTA «Оплатить консультацию» (например, в письме или на странице после отправки формы).

---

### Сценарий B: Покупка курса (пользователь уже зарегистрирован)

1. Пользователь на главной нажимает «Купить» по тарифу с курсом → модалка → оплата на PayKeeper.
2. Webhook: order paid, user найден по email → enrollment, уведомление, письмо.
3. Пользователь заходит в ЛК → «Мои курсы» → курс доступен.
4. **Сейчас:** реализовано.

---

### Сценарий C: Покупка курса (первый раз, без аккаунта)

1. Гость на главной покупает курс → оплата → webhook: user не найден, отправляется письмо «зарегистрируйтесь с этим email».
2. Пользователь переходит по ссылке из письма или с /success → регистрируется с тем же email.
3. **Ожидание:** после регистрации курс сразу виден в «Моих курсах».
4. **Сейчас:** после регистрации enrollment не создаётся, курс не появляется. Нужна привязка оплаченных заказов при регистрации.

---

### Сценарий D: Покупка консультации / тренинга (без курса)

1. Гость или пользователь оплачивает тариф «Консультация» / «Тренинг» / «Онлайн».
2. Webhook: order paid, у тарифа нет courseId → только обновление заказа.
3. **Ожидание:** клиенту приходит письмо «Оплата получена, мы свяжемся с вами» или «Ожидайте подтверждения времени».
4. **Сейчас:** письмо не отправляется.

---

### Сценарий E: Конвертация лида в пользователя

1. Админ в CRM конвертирует лид → создаётся User с случайным паролем.
2. **Ожидание:** клиенту уходит письмо с ссылкой «Установить пароль» или временный пароль и ссылка на вход.
3. **Сейчас:** клиенту ничего не отправляется; админ должен вручную сообщить пароль или отправить ссылку на сброс (которая в локальном режиме недоступна).

---

### Сценарий F: Успешная оплата → страница /success

1. После оплаты пользователь попадает на /success (редирект PayKeeper).
2. **Ожидание (опционально):** для авторизованного — «Ваш курс „…“ уже в разделе „Мои курсы“»; для гостя — «Зарегистрируйтесь с email … чтобы увидеть курс».
3. **Сейчас:** общий текст и кнопки без персонализации.

---

### Сценарий G: Первый вход после регистрации / покупки

1. Пользователь впервые заходит в ЛК (после регистрации или после конвертации лида).
2. **Ожидание:** приветственное уведомление/письмо, краткий онбординг (где курсы, где поддержка).
3. **Сейчас:** нет welcome-письма и онбординга.

---

### Сценарий H: Автоподдержка (триггерные действия)

1. **Ожидание примеров:**  
   - после оплаты курса без аккаунта — письмо с явной ссылкой «Зарегистрироваться и открыть курс»;  
   - после регистрации — письмо «Как пользоваться ЛК»;  
   - при долгом отсутствии в курсе (N дней) — напоминание (если нужно бизнесу).  
2. **Сейчас:** тикеты только ручные; авто-тикеты и триггерные письма не реализованы.

## 3. План доработок (блоки и функционал)

|---|--------|-----------|----------|
| 1.1 | Письмо клиенту «Заявка принята» | Высокий | После POST `/api/contact` отправлять клиенту письмо (если указан email): «Мы получили вашу заявку, свяжемся в течение …» |
| 1.2 | Страница/блок «Заявка отправлена» | Средний | После отправки формы показывать экран «Спасибо, заявка принята» с CTA «Оплатить консультацию» (ссылка на #pricing или модалка оплаты) |
| 1.3 | Связь Lead ↔ Order (опционально) | Низкий | При создании заказа по email, совпадающему с лидом, проставлять в Lead ссылку на заказ или помечать «оплатил» для аналитики |

### Блок 2: Оплата и webhook

| № | Задача | Приоритет | Описание |
|---|--------|-----------|----------|
| 2.1 | Письмо при оплате тарифа без курса | Высокий | В webhook PayKeeper: если у заказа нет привязки к курсу (consult/group/online), всё равно отправлять клиенту письмо «Оплата получена. Мы свяжемся с вами» (или шаблон из настроек) |
| 2.2 | Единый шаблон писем об оплате | Средний | Вынести тексты писем об оплате (с курсом / без курса) в шаблоны уведомлений или в SystemSetting, чтобы можно было править без кода |

### Блок 3: Авторегистрация и привязка заказа

| № | Задача | Приоритет | Описание |
|---|--------|-----------|----------|
| 3.1 | Привязка оплаченных заказов при регистрации | Критический | В POST `/api/auth/register` после создания User: найти заказы `Order` где `clientEmail = email`, `status = 'paid'`, `userId = null`; для каждого заказа с курсом (через Service) создать Enrollment, вызвать triggerNotification('enrollment'), проставить Order.userId |
| 3.2 | То же при первом входе (страховка) | Средний | При логине (или в middleware/layout студента): если у пользователя нет enrollment, но есть оплаченные заказы по его email — выполнить привязку (как в 3.1). Защита от дублей по уже заполненному Order.userId |

| № | Задача | Приоритет | Описание |

| 4.2 | Query-параметр order | Низкий | Поддержать ?order=ORDER_NUMBER (если PayKeeper может вернуть): для гостя показывать «Заказ … оплачен. Зарегистрируйтесь с email …» (email брать из Order по orderNumber, если есть и если безопасно) |

### Блок 5: Конвертация лида и автоподдержка клиента

| № | Задача | Приоритет | Описание |
|---|--------|-----------|----------|
| 5.1 | Письмо при конвертации лида | Высокий | После конвертации лида в пользователя: отправлять клиенту письмо со ссылкой «Установить пароль» (страница /auth/set-password?token=…). Реализовать одноразовый токен (таблица или поле в User/Profile), срок жизни 24–48 ч |
| 5.2 | Страница «Установить пароль» по токену | Высокий | Маршрут /auth/set-password: по токену определить пользователя, форма «Новый пароль» + подтверждение; после установки — инвалидация токена, редирект на /login |
| 5.3 | Альтернатива: письмо с временным паролем | Средний | Если не делать set-password по ссылке: в письме при конвертации указывать временный пароль и «Рекомендуем сменить после первого входа» (минус: пароль в почте) |
