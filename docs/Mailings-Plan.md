# План разработки модуля «Рассылки»

Документ составлен на основе мини-ТЗ (инструкция [1–97], структура системы) и промта для Cursor. Модуль предназначен для массовых email-рассылок с журналом результатов и планировщиком.

---

## 1. Назначение модуля

Проведение **массовых рассылок информационных сообщений** пользователям системы на электронную почту: создание рассылки (тема, текст, адресаты), отправка вручную или по расписанию, журнал по каждому получателю (успех/ошибка).

---

## 2. Базовые сущности

| Сущность | Описание |
|----------|----------|
| **Рассылка (Mailing)** | Объект с темой, телом письма, списком адресатов, настройками отправки (режим, время), статусом. |
| **Журнал рассылки (MailingLog)** | Записи о результате отправки по каждому адресату: отправлено / ошибка, текст ошибки, время. |

---

## 3. Ключевые поля рассылки (MVP)

| Поле | Тип | Описание |
|------|-----|----------|
| **Название** | string | Внутреннее имя рассылки (для каталога в админке). |
| **Тема** | string | Заголовок письма (Subject), который видит получатель. |
| **Текст** | string (HTML) | Тело письма с поддержкой HTML и подстановки ключевых слов. |
| **Отправитель** | — | ФИО и e-mail, от имени которых отправляется письмо (поля senderName, senderEmail). |
| **Режим отправки** | enum | «В ручном режиме» (manual) — по нажатию «Отправить»; «В указанное время» (scheduled) — по расписанию. |
| **Запланировано на** | datetime | Дата и время отправки (при режиме scheduled). |
| **Статус** | enum | Запланирована (planned), Идет (processing), Завершена (completed). |
| **Вложения** | — | Список прикреплённых файлов (пути или ключи в хранилище); контроль максимального объёма (МБ). |

## 4. Функциональные требования

### 4.1. Шаблонизация (ключевые слова)

При отправке в теле и теме письма заменять плейсхолдеры на данные пользователя и системы:

| Плейсхолдер | Подстановка |
|-------------|-------------|
| `%FirstName%` | Имя пользователя (displayName или часть до пробела / из email). |
| `%LastName%` | Фамилия или вторая часть ФИО (при наличии). |
| `%date%` | Текущая дата (локаль или ISO). |
| `%unsubscribe%` | Ссылка для отписки от рассылок (обязательна для соблюдения правил; генерировать реальный URL или заглушку). |

**Шаблон в стиле сайта:** в `lib/email-templates.ts` заданы `DEFAULT_MAILING_BODY`, `DEFAULT_MAILING_SUBJECT` и функция `renderMailingTemplate`. Тело письма рассылки рекомендуется оборачивать через `wrapEmailHtml()` для единого оформления AVATERRA (шапка, цвета, подвал).

### 4.2. Управление адресатами

- **Вкладка «Включаемые адресаты»:** выбор конкретных пользователей (по списку) и/или групп (например, по роли: все студенты, все менеджеры).
- **Итоговый список:** динамическое формирование перед отправкой (объединение выбранных пользователей, исключение дублей, фильтр «только с указанным e-mail»).

### 4.3. Процесс отправки

- Рассылка выполняется **только один раз**. Повторная отправка той же рассылки недоступна.
- Для повтора — функция **«Копировать»**: создаётся копия рассылки со сброшенным статусом в «Запланирована», без логов; копию можно редактировать и отправить.
- **Вложения:** возможность прикрепить файлы к рассылке; проверка суммарного размера (например, макс. 10 МБ).

### 4.4. Контроль результатов

- **Журнал рассылки:** по каждому получателю — статус (Отправлено / Ошибка), при ошибке — сообщение (например, «не указан e-mail», «ошибка SMTP»).
- **Сводка:** время начала и окончания, всего отправлено, всего ошибок.

## 5. Модель данных (Prisma)

### 5.1. Рассылка (Mailing)

```prisma
model Mailing {
  id            String    @id @default(cuid())
  internalTitle String    // внутреннее название
  emailSubject  String
  emailBody     String    // HTML
  senderName    String?
  senderEmail   String?
  status        String    @default("planned")  // planned | processing | completed
  scheduleMode  String    @default("manual")  // manual | scheduled
  scheduledAt   DateTime?
  attachments   String?   // JSON: массив путей/ключей вложений
  createdById   String?
  createdBy     User?     @relation(fields: [createdById], references: [id], onDelete: SetNull)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  startedAt     DateTime?
  completedAt   DateTime?

  logs          MailingLog[]
}
```

### 5.2. Журнал рассылки (MailingLog)

```prisma
model MailingLog {
  id          String   @id @default(cuid())
  mailingId   String
  mailing     Mailing  @relation(fields: [mailingId], references: [id], onDelete: Cascade)
  userId      String?
  user        User?    @relation(fields: [userId], references: [id], onDelete: SetNull)
  recipientEmail String
  recipientName  String?
  status      String   // "sent" | "failed"
  errorMessage String?
  sentAt      DateTime?
  createdAt   DateTime @default(now())

  @@index([mailingId])
}
```

- В модели `User` добавить связи: `mailingsCreated Mailing[]`, `mailingLogs MailingLog[]` (при необходимости).

### 5.3. Ограничение вложений

- Хранение: в `attachments` — JSON-массив объектов `{ name, pathOrKey, size }`.
- При сохранении рассылки и при отправке проверять: суммарный размер ≤ заданного лимита (например, 10 МБ).

## 6. Логика работы

### 6.1. Статусы

| Статус | Описание |
|--------|----------|
| **Запланирована (planned)** | Рассылка создана, можно редактировать; доступны «Отправить сейчас» или ожидание запланированного времени. |
| **Идет (processing)** | Отправка выполняется (фоново или синхронно). |
| **Завершена (completed)** | Отправка выполнена; редактирование и повторная отправка недоступны, доступно только «Копировать». |

### 6.2. Режимы отправки

- **manual:** отправка только по нажатию «Отправить сейчас» в админке.
- **scheduled:** отправка в момент `scheduledAt` (нужен планировщик: cron, очередь или проверка по запросу).

### 6.3. Подстановка ключевых слов

- Перед отправкой письма каждому получателю: заменить в `emailSubject` и `emailBody` плейсхолдеры на данные этого пользователя и текущую дату.
- **%unsubscribe%** — подставлять ссылку на страницу отписки (например, `/unsubscribe?token=...` или заглушка `/unsubscribe`); в MVP достаточно единой ссылки или текста «Отписаться от рассылок».

### 6.4. Ошибки и журнал

- Если у пользователя нет e-mail — не отправлять, записать в MailingLog статус `failed`, `errorMessage`: «Не указан e-mail».
- Если ошибка отправки (SMTP, Resend API) — записать в MailingLog статус `failed` и текст ошибки.
- Успешная отправка — запись со статусом `sent`, `sentAt`.

## 7. Админ-интерфейс

### 7.1. Список рассылок

- Таблица: [№], Название, Создатель, Статус, Режим (Ручной / По расписанию), Дата создания, Действия (Просмотр / Копировать / Удалить).
- Удаление только для рассылок в статусе «Запланирована» или с ограничением по политике (например, разрешить удаление с предупреждением).

### 7.2. Форма создания/редактирования

- **Вкладка «Основное»:** название, тема письма, тело (rich-text / textarea с подсказкой про HTML и ключевые слова), ФИО и e-mail отправителя, режим отправки (ручной / по расписанию), при режиме «по расписанию» — дата и время, загрузка вложений с отображением размера и лимита.
- **Вкладка «Адресаты»:** выбор получателей — по списку пользователей (множественный выбор) и/или по группе (роль). Итоговый список: предпросмотр количества получателей (с e-mail) перед отправкой.
- **Действия:** «Сохранить», «Отправить сейчас» (только для статуса «Запланирована» и при ручном режиме), «Копировать» (на странице просмотра завершённой рассылки).

### 7.3. Просмотр рассылки и журнал

- Карточка рассылки: все поля, статус, даты начала/окончания.
- **Вкладка «Результаты»:** сводка (время начала, окончания, всего отправлено, всего ошибок), таблица журнала: Получатель (имя), E-mail, Статус (Отправлено / Ошибка), Сообщение об ошибке, Время.

## 8. API

| Метод | Маршрут | Назначение |
|-------|---------|------------|
| GET | `/api/portal/admin/mailings` | Список рассылок для админки. |
| POST | `/api/portal/admin/mailings` | Создание рассылки (черновик, статус planned). |
| GET | `/api/portal/admin/mailings/[id]` | Одна рассылка с логами (для просмотра и результатов). |
| PATCH | `/api/portal/admin/mailings/[id]` | Обновление рассылки (только если статус planned). |
| DELETE | `/api/portal/admin/mailings/[id]` | Удаление (только planned или с ограничениями). |
| POST | `/api/portal/admin/mailings/[id]/send` | Запуск отправки «сейчас» (только planned, manual). |
| POST | `/api/portal/admin/mailings/[id]/copy` | Копирование рассылки (новый id, статус planned, без логов). |
| GET | `/api/portal/admin/mailings/[id]/recipients-preview` | Предпросмотр итогового списка адресатов (количество, список email). |

Для запланированной отправки в указанное время: либо cron-задача, либо API «тик» планировщика, который находит рассылки с `scheduleMode=scheduled`, `scheduledAt <= now`, `status=planned` и запускает отправку.

### Этап 1. Модель данных и миграции

- [ ] Валидация вложений: суммарный размер ≤ лимита (например, 10 МБ).

### Этап 3. Админ-интерфейс: список и форма

- [ ] Страница «Рассылки» в админке: таблица (Название, Создатель, Статус, Режим, Действия).
- [ ] Форма создания/редактирования: вкладки «Основное» (название, тема, тело, отправитель, режим, дата, вложения) и «Адресаты» (выбор пользователей и/или по роли).
- [ ] Кнопки «Сохранить», «Отправить сейчас» (видна только для planned + manual).
- [ ] Подсказка по ключевым словам: %FirstName%, %LastName%, %date%, %unsubscribe%.

### Этап 4. Логика отправки и подстановка

- [ ] Функция подстановки ключевых слов в тему и тело (по данным пользователя и дате).
- [ ] Генерация ссылки отписки %unsubscribe% (страница /unsubscribe или заглушка).
- [ ] При нажатии «Отправить сейчас»: формирование итогового списка получателей (с e-mail), переход в processing, отправка писем (Resend или существующий sendEmail), запись в MailingLog по каждому; по окончании — status=completed, completedAt.
- [ ] Обработка ошибок: нет e-mail, ошибка API — фиксация в MailingLog с errorMessage.

### Этап 5. Журнал и результаты

- [ ] После отправки: заполнение MailingLog (recipientEmail, recipientName, status, errorMessage, sentAt).
- [ ] В админке: вкладка/блок «Результаты» у рассылки — сводка (время начала/окончания, всего отправлено, всего ошибок) и таблица журнала (Получатель, E-mail, Статус, Ошибка, Время).

### Этап 6. Копирование и ограничения

- [ ] API «Копировать»: клонирование рассылки (без логов, status=planned), возврат id новой рассылки; редирект на редактирование новой рассылки.
- [ ] Запрет редактирования и повторной отправки для рассылок со статусом completed.
- [ ] Удаление: только для planned или с предупреждением (в т.ч. удаление логов при удалении рассылки).

### Этап 7. Планировщик (отправка в указанное время)

- [ ] Реализация проверки запланированных рассылок: scheduledMode=scheduled, scheduledAt <= now(), status=planned. Варианты: cron-скрипт, API route по крону, очередь (если есть).
- [ ] Запуск той же логики отправки, что и для «Отправить сейчас».

### Этап 8. Отписка и полировка

- [ ] Страница/маршрут отписки (например, `/unsubscribe?token=...`): сохранение предпочтения пользователя (не присылать рассылки); подстановка ссылки в %unsubscribe%.
- [ ] Стили: светлый фон, белые карточки, акцентные кнопки (primary/gold). Документация (Support.md, Tasktracker).

## 11. Критерии приёмки (MVP)

- В админке есть каталог рассылок с созданием, редактированием (только для «Запланирована»), удалением, просмотром.
- Форма рассылки: название, тема, тело (HTML), отправитель (ФИО, e-mail), режим (ручной / по расписанию), дата/время при scheduled, вложения с контролем размера, вкладка «Адресаты» (выбор пользователей и/или по роли).
- Подстановка при отправке: %FirstName%, %LastName%, %date%, %unsubscribe% (ссылка отписки обязательна).
- Рассылка выполняется один раз; для повтора — кнопка «Копировать» (новая рассылка со статусом «Запланирована»).
- Журнал рассылки: по каждому получателю статус (Отправлено / Ошибка) и сообщение об ошибке (в т.ч. «не указан e-mail»); сводка: время начала/окончания, всего отправлено, всего ошибок.
- Кнопка «Отправить сейчас» запускает отправку для рассылки в статусе «Запланирована» (ручной режим).
- Запланированная отправка в указанное время реализована (cron или аналог).

## 12. Связь с существующим кодом

- **Отправка писем:** использовать существующий механизм (например, `lib/email.ts`, Resend) для отправки одного письма; в модуле рассылок — цикл по адресатам и запись в MailingLog.
- **Коммуникации (comms):** модуль «Коммуникации» — шаблоны и разовые рассылки по шаблону; модуль «Рассылки» — отдельные сущности Mailing/MailingLog с собственным жизненным циклом, планировщиком и журналом. При необходимости оба могут использовать общий `sendEmail`.

## 13. Рекомендации из источников

- **Копирование** — обязательная функция: завершённую рассылку нельзя редактировать или отправлять повторно; только копия с обнулённым статусом.
- **Статусы ошибок** — фиксировать в журнале случаи «нет e-mail у получателя» и «ошибка отправителя/SMTP» как «Замечания при выполнении».
- **Отписка** — обязательно предусмотреть подстановку ссылки %unsubscribe% для соблюдения правил рассылок (реальная страница отписки или заглушка с последующей доработкой).
