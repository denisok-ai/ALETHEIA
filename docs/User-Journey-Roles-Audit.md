# Аудит пользовательских путей по ролям

**Дата:** 2026-03-12  
**Цель:** убедиться, что весь функционал логичен, удобен и отображён в интерфейсе для ролей **Слушатель (user)**, **Менеджер (manager)**, **Администратор (admin)**.

---

## 1. Роли и точки входа

| Роль       | Роль в коде | Точка входа после входа | Разделы портала        |
|-----------|-------------|--------------------------|-------------------------|
| Слушатель | `user`      | ЛК студента              | `/portal/student/*`     |
| Менеджер  | `manager`   | Кабинет менеджера        | `/portal/manager/*`     |
| Админ     | `admin`     | Админка                  | `/portal/admin/*`       |

**Защита:** middleware проверяет роль для `/portal/admin` (только admin) и `/portal/manager` (manager или admin). Студент не может зайти в админку и кабинет менеджера.

## 3. Выявленные пробелы и рекомендации

### 3.1 Редирект после входа и при открытии /portal

| Проблема | Было | Рекомендация |
|----------|------|--------------|
| После логина все попадают в ЛК студента | `redirect ?? '/portal/student/dashboard'` | Редирект по роли: admin → `/portal/admin/dashboard`, manager → `/portal/manager/dashboard`, user → `/portal/student/dashboard`. |
| Открытие /portal | Нет страницы, возможна 404 | Сделать `/portal` редиректом на нужный дашборд по роли. |

**Реализация:** страница `app/portal/page.tsx` с server-side редиректом по сессии; на странице входа по умолчанию использовать `redirect=/portal`.

### 3.2 Переключение между разделами (admin/manager)

| Проблема | Описание |
|----------|----------|
| Админ заходит по умолчанию в ЛК студента | После правки п. 3.1 будет попадать в админку. Если зайдёт, например, по прямой ссылке в ЛК — нужен быстрый переход «В админку». |
| Менеджер с ролью manager | Аналогично: из ЛК студента — переход «В кабинет менеджера». |
| Админ в кабинете менеджера | Переход «В админку» и при необходимости «В ЛК студента». |

**Рекомендация:** в блоке учётной записи в сайдбаре (PortalAccountBlock) показывать ссылки переключения по ролям:
- для **admin**: «Админка» (если сейчас не в админке), «Кабинет менеджера», «ЛК студента»;
- для **manager**: «Кабинет менеджера» (если не в нём), «ЛК студента».

Так весь функционал остаётся видимым и доступным без запоминания URL.

### 3.3 Уведомления для менеджера и админа

В PortalHeader (если будет использоваться) колокольчик для роли `user` ведёт в «Уведомления», для manager/admin — в `#`. Рекомендация: для manager/admin либо скрыть колокольчик, либо вести на осмысленную страницу (например, журнал уведомлений для админа, тикеты для менеджера).

### 3.4 Профиль менеджера

«Мой профиль» для менеджера ведёт на дашборд. При необходимости можно добавить страницу `/portal/manager/profile` (редактирование имени, email и т.д.) по аналогии с студентом.

### 3.5 Соответствие Spec и видимость в UI

| Блок из Spec | В интерфейсе |
|--------------|--------------|
| ЛК студента: дашборд, курсы, SCORM, сертификаты, медиатека, уведомления, поддержка, профиль | ✅ Все пункты в сайдбаре и на страницах. |
| Кабинет менеджера: тикеты, пользователи, верификация | ✅ В сайдбаре и на страницах. |
| Админка: пользователи, курсы, CRM, оплаты, медиа, коммуникации, AI, аудит, сертификаты и т.д. | ✅ В сайдбаре по секциям; вложенные сущности — по ссылкам из таблиц/карточек. |

Скрытого функционала, который есть в коде, но недоступен из UI, не выявлено.

## 5. Внесённые изменения

1. **Сделано:** страница `app/portal/page.tsx` — при открытии `/portal` выполняется редирект на дашборд по роли (admin → админка, manager → кабинет менеджера, user → ЛК студента).
2. **Сделано:** на странице входа по умолчанию `redirect=/portal` вместо `/portal/student/dashboard`, чтобы после логина срабатывал редирект по роли.
3. **Сделано:** в блоке учётной записи сайдбара (PortalAccountBlock) для ролей admin и manager добавлены ссылки переключения раздела: «Админка», «Кабинет менеджера», «ЛК студента». Отображаются только те ссылки, на которые пользователь имеет право и которые ведут в другой раздел (текущий не показывается).

После этих правок пользовательские пути для всех ролей становятся предсказуемыми, а весь функционал остаётся отображённым и доступным из интерфейса.
